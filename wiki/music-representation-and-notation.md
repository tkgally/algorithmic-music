# Music representation and notation

Every generative engine commits, explicitly or by default, to a data structure standing in for the piece before or instead of sound, and that choice quietly bounds what the engine can express — a schema with no field for microtiming cannot produce groove, one with no field for cents cannot produce a raga. This page surveys how humans have notated music on paper (staff notation, tablature, functional shorthand, cipher systems) and how machines encode it (MIDI and its descendants, notation-interchange XML, compact text, tracker patterns, control protocols), draws out what a generative engine's internal representation needs from both traditions, and ends with practical guidance for packing that representation into a shareable URL — the concrete gap this page exists to fill for [comprehensive-site-vision](comprehensive-site-vision.md)'s compact-state requirement.

## Why representation is a design decision, not a detail

[engine-architecture](engine-architecture.md) already separates style pack, planner, composer, performer, and synthesizer into a one-directional pipeline precisely because each stage wants a different representation of the same piece (declarative rules → a form plan → a symbolic score in beats → an expressive rendering in seconds → an audio graph); the schemas at each boundary are the actual interface between stages, and getting them wrong compounds downstream. This page treats those boundary schemas as a representation-design problem with a long prior literature — human notation and machine formats have been solving overlapping versions of it for centuries — rather than reinventing the choices from nothing.

## Human notation: what each system captures, and what it discards

No human notation records "the music" whole; each is a deliberate, lossy projection tuned to a use case, and the specific losses are informative for engine design (a representation that keeps only what its target notation kept will reproduce that notation's blind spots).

| System | Fixes precisely | Leaves out or assumes |
|---|---|---|
| Staff notation | Pitch (line/space + clef + accidentals) and rhythm (note values) to fine grain | Timbre, microtiming, and continuous expression; dynamics/articulation are coarse categorical marks, not curves |
| Tablature (e.g., guitar TAB; guqin *jianzipu* — [east-asian-traditions](east-asian-traditions.md)) | The physical action: string, fret/position, finger or technique | Often rhythm entirely (guqin's *jianzipu* omits it, recovered by the *dapu* [打譜, "reconstructing the notation"] reading tradition); always instrument- and tuning-specific pitch |
| Figured bass | Chord and inversion relative to a written bass note, via numerals (5/3 unmarked, 6, 6/4, 7, 6/5, 4/3, 4/2) | Voicing, doubling, register, ornamentation — left to the continuo player's judgment |
| Nashville Number System | Chord root and quality as a scale-degree number relative to the current key | Voicing and register entirely; rhythm beyond a slash or hold mark |
| Lead sheet | Melody (staff) plus chord symbols (plus lyrics) | The accompaniment texture, improvised live from the symbols |
| TUBS / box notation | Onset timing only, one row per part on a shared time grid | Pitch, dynamics, timbre — unless a variant adds symbols for them |

Two entries reward more discussion because they encode harmony as a *transposable function* rather than fixed pitches, which is exactly the property a style pack's chord-transition table wants ([style-and-genre-overview](style-and-genre-overview.md), [harmony](harmony.md)): figured bass numerals describe a chord's relationship to its bass regardless of key, and the Nashville Number System (commonly credited to session bassist Neal Matthews Jr., late-1950s Nashville — [blues-country-and-roots](blues-country-and-roots.md) is its home genre context) writes "1 4 5 1" instead of "C F G C" so the same chart transposes for free between vocalists. Both are centuries- and decades-old proof that "store harmony as scale-degree function, resolve to pitch only at the end" scales to real working repertoires, not just to theory examples.

Beyond the table, three cipher and gesture systems outside the Western staff tradition, one line each: **jianpu** [简谱, "simplified notation"], China — movable-1 digits 1–7 for scale degrees (1 = the current tonic), octave dots above/below, underlines to halve duration, tracing to Rousseau's 1742 numbered-notation proposal, systematized in 19th-century France as the Galin-Paris-Chevé method, and adopted in China by 1900 ([east-asian-traditions](east-asian-traditions.md)); **sargam**, North and South Indian classical music — the solfège syllables Sa Re Ga Ma Pa Dha Ni with *komal*/*tivra* [कोमल/तीव्र, "flattened"/"sharpened"] variants marking altered scale positions, movable like jianpu (Sa is fixed only relative to itself, not to a frequency) — [indian-classical-music](indian-classical-music.md); **kepatihan**, Central Javanese gamelan — cipher numbers for the *balungan* ("skeleton", the core melody — same term and gloss as [gamelan](gamelan.md)), 1–7 for *pélog* and 1,2,3,5,6 for *sléndro*, plus symbols for the colotomic gong punctuation, devised around 1900 at the Kepatihan (the Grand Vizier's compound) in Surakarta from the same Galin-Paris-Chevé cipher lineage as jianpu, imported via missionary hymn notation. Both cipher traditions independently converge on "movable scale-degree digits plus octave marks" as the compact, typeable, monophonic-friendly answer — worth noting as one candidate shape for any lightweight internal melody-debug format this project might want.

## Machine formats

| Format | Data model | Timing model | Where it is strong |
|---|---|---|---|
| MIDI (protocol) | Per-channel status/data byte stream: note on/off, control change, pitch bend | None — a live/streamed protocol, not a file | Real-time device and software control |
| Standard MIDI File (SMF) | Multi-track chunked binary of MIDI-derived events | Integer ticks (PPQ) with delta-times, or SMPTE frames | Lossless interchange of a MIDI performance or arrangement |
| MusicXML | XML tree: parts → measures → notes, with layout hints | Measure- and note-duration-based | Interchange between notation *editors* (Finale, Sibelius, MuseScore) |
| MEI | XML tree, structured extensible semantics | As MusicXML, plus source/facsimile linkage | Scholarly encoding; the one format with native mensural and neume support |
| ABC | Plain-text, line-oriented tune format | Note-length tokens relative to a declared default | Compact, human-typable monophonic melody plus chords |
| Tracker / MOD | Pattern = channel columns × row grid; song = a sequence of pattern indices | One row = one fixed time slice | Algorithmically generated or hand-edited step patterns |
| OSC | URL-style address pattern + typed argument list | None — a live message protocol | Low-latency inter-process/inter-device control, not storage |

**MIDI's note model and its limits.** A MIDI message is a status byte (high bit 1, low nibble = one of 16 channels) followed by one or two 7-bit data bytes (high bit 0, so 0–127) (MIDI Association). Note On (`0x9n`) carries a note number and a velocity, both 0–127 — 128 pitches, 128 velocity steps, and (per this wiki's convention) note 60 = middle C = C4. The design decision that shapes everything downstream is that expression — pitch bend, mod wheel, aftertouch — is scoped **per channel**, not per note: one pitch-bend value bends every currently-sounding note on that channel identically, and General MIDI's channel-10-for-percussion convention shows how quickly 16 channels run out for anything but simple arrangements.

**MPE and MIDI 2.0 make expression per-note.** MIDI Polyphonic Expression (MMA/AMEI Recommended Practice, v1.0, 2018-03-12) works around the per-channel ceiling without changing the wire protocol: a "zone" reserves one master channel for global messages and a block of member channels, one per currently-sounding note, so each note's pitch bend (member channels default to a ±48-semitone range, versus ±2 for the master), channel pressure, and CC74 ("timbre") apply independently — at the cost of roughly 15 simultaneous fully expressive notes per 16-channel stream (MIDI Association). MIDI 2.0 (announced 2020-01-17) removes the workaround: its Universal MIDI Packet format (32/64/96/128-bit words, 16 groups of 16 channels, backward-compatible — a group can carry a plain MIDI 1.0 stream) adds genuine per-note pitch, controllers, and attribute fields alongside far higher resolution, commonly reported as 16-bit velocity (65,536 steps versus 128) and 32-bit pitch bend/controllers/pressure, with per-note pitch reportedly resolvable to a small fraction of a cent (~1/512 semitone) — comfortably enough for any microtonal system this wiki documents, though that specific figure is web-secondary-sourced here, not checked against primary AMEI spec text (Wikipedia, "MIDI 2.0"; MIDI Association). None of this reaches this project directly today — engines are self-contained browser code with no MIDI I/O — but it is the right mental model for "per-note continuous parameters," which is exactly what [engine-architecture](engine-architecture.md)'s Note event schema and this project's shipped engines have converged on independently (below).

**Standard MIDI Files store ticks, not seconds.** An SMF header declares a division: either pulses/ticks per quarter note (PPQN — 24 and 96 are historic low-resolution values, 480 and 960 common today) or an SMPTE-frame timing mode; every event's delta-time is then a variable-length quantity — 7 payload bits per byte, high bit as a continuation flag, so small gaps cost one byte and only large ones grow — and "running status" lets consecutive same-status events omit repeated status bytes entirely. Three format variants exist: 0 (one merged track), 1 (multiple simultaneous tracks), and 2 (multiple independent sequential tracks — closer to a tracker's song-of-patterns model) (SMF spec). SMF's integer-tick discipline is the same rationale [scheduling-and-timing](scheduling-and-timing.md) gives for why this project keeps musical position as an integer/exact value rather than repeatedly accumulating floating seconds: it is a lossless-interchange time base, not covered further here.

**MusicXML and MEI split rendering from semantics.** MusicXML was built as an interchange format between notation *editors* — its XML tree of parts, measures, and notes carries layout hints (beam grouping, stem direction) because round-tripping through engraving software is the point. MEI targets the same notational content but is designed to capture semantics and scholarly intent as structured, extensible metadata, which is also why it is the one of the two with native support for pre-modern notations (mensural, neume): it models notation as structured meaning rather than fixed engraving primitives (Open Sheet Music Display; Music Encoding Initiative). Neither is a runtime target here — XML parsing with no build step buys nothing an engine needs, and the content (score images) isn't the deliverable — but MEI's "semantics over rendering" choice is the same principle [engine-architecture](engine-architecture.md)'s Note event `tags?` field already makes at far smaller scale (analytical intent riding alongside the note, not just enough to draw it). Both formats' role as symbolic-corpus sources is [corpus-analysis](corpus-analysis.md)'s territory, not repeated here.

**ABC and tracker/MOD: two compact, generation-friendly text/grid formats.** ABC is plain ASCII — uppercase C–B for one octave starting at middle C (this wiki's C4), lowercase c–b for the octave above, commas/apostrophes to shift further, digits and slashes for duration, header fields (`X:`, `T:`, `K:`, `M:`) for tune metadata (current standard v2.1, December 2011, Chris Walshaw). Its relevant lesson here isn't corpus use ([corpus-analysis](corpus-analysis.md) already covers that) but the design proof that a monophonic melody plus a chord-symbol line compresses to a few dozen bytes of readable text — a useful gut check for how compact a symbolic phrase can be even before bit-packing. Tracker/MOD formats, established by Karsten Obarski's Ultimate Soundtracker (1987, Amiga, four channels) and extended by NoiseTracker and ProTracker, represent a song as a sequence of *patterns*, each a fixed-length table with one column per channel and one row per time step, cells holding a note plus instrument plus one effect command (Wikipedia, "Module file"). That pattern grid is a piano-roll-shaped matrix with slots practically begging to be filled algorithmically — exactly why demoscene and chiptune generative work grew up on trackers — and it is structurally the same idea as TUBS (Philip Harland and James Koetting, early 1960s, for notating African polyrhythm: one row per part, boxes for onsets) rotated ninety degrees, a direct lineage from ethnomusicological box notation to the step-sequencer grid every drum machine still uses (Wikipedia, "Time unit box system"; [west-african-rhythm](west-african-rhythm.md)).

**OSC, briefly.** Open Sound Control addresses a receiver with a URL-style pattern (e.g., `/synth/1/freq`) carrying a typed argument list (int32, float32, string, blob, via a type-tag string), conventionally over UDP; CNMAT (Matt Wright and Adrian Freed) published the first spec in 2002, with a 1.1 revision in 2009 (Wikipedia, "Open Sound Control"). It is a live control protocol, not a storage format, and it is not directly usable here regardless — browsers expose no raw UDP sockets, and the project's hard rule against network calls at runtime would rule it out even if they did. Its address-pattern-plus-typed-argument idea is nonetheless a reasonable model for naming this project's own in-process parameters hierarchically (`reverb.wet`, `lead.vibrato.depth`), a question for [control-surfaces-and-user-parameters](control-surfaces-and-user-parameters.md) rather than this page.

## Representations for generation

**Event lists, piano-roll matrices, and hierarchical trees.** An event list — a flat array of `{onset, duration, pitch, …}` objects, this project's Note event — is sparse, needs no fixed time quantum, and is the natural fit for arbitrary polyphony, but carries no innate structure: a phrase or a bar is not represented, only reconstructable. A piano-roll matrix (fixed time-step × pitch grid) is the opposite trade: trivial to reason about "what's sounding when" and a natural fit for grid-based tooling, but forces a time-quantization decision up front, wastes cells at low note density, and (per a recent survey of deep symbolic generation) struggles to distinguish a held note from a repeated short one without extra channels, while pure event streams are compact but brittle to edit — inserting or deleting a note can force re-tokenizing everything after it (Ji, Luo & Yang 2020). A hierarchical tree — a grammar derivation or a GTTM-style time-span/prolongational reduction — sits above either, encoding *why* an event exists (structural tone versus elaboration); [grammars-and-rewriting-systems](grammars-and-rewriting-systems.md) covers GTTM and Schenkerian reduction in depth and is not repeated here. The representational point specific to this page: a tree is an annotation layer over the event list, not a replacement for it — an event needs only a pointer into its derivation (which `tags?` already approximates cheaply), and any future [meta-composition-and-style-machines](meta-composition-and-style-machines.md) system that generates *at the level of rules or grammars themselves* still needs a plain event list as its eventual output, because that is what the performer, the offline-render harness, and the metrics layer all consume ([computational-music-metrics](computational-music-metrics.md)).

**Per-note parameter objects beat fixed fields.** [engine-architecture](engine-architecture.md)'s provisional schemas — Note event `{ beat, durBeats, midi, voice, role, artic?, tags? }`, Performance event `{ timeSec, durSec, freqHz, vel, voice, artic, detuneCents? }` — already point the right way with an open `tags?` field, but "provisional" undersells how far shipped practice has run with the idea. Reading the five shipped engines' code directly: every one emits Performance events as `{ timeSec, durSec, freq, vel, voice, beat, midi, tags, … }` — `freq`, not `freqHz` as documented, and with `beat`/`midi`/`tags` carried straight through from the Note event rather than dropped at the beat→seconds boundary, so symbolic provenance survives into the rendered event for free (verified in `docs/preliminary-tests/engines/{01,02,03,04,05}-*/engine.js`). Engine 04 (cantabile) goes further: every note optionally carries `weight` (a composer-stamped 0–1 structural-importance scalar) and `expr`, a nested object of roughly fifteen continuous-shape parameters — onset/release glide in cents, vibrato rate/depth/delay/ramp, swell amount and peak position, brightness, grit, breath grain, micro-pitch drift, attack/release time (`docs/preliminary-tests/engines/04-cantabile/engine.js`) — that the performer reads to drive intra-note synthesis, exactly the "note with an inner life" [findings-cantabile-engine](findings-cantabile-engine.md) describes. The general lesson: a fixed field list is a MIDI-1.0-shaped assumption (pitch, duration, velocity, and stop); the moment an engine wants a new expressive dimension, a fixed schema forces either overloading an existing field or a breaking change for every consumer. An open, nested, all-optional object absorbs new dimensions additively — new engines add fields, old renders ignore fields they don't recognize, nothing upstream breaks — the same trick MIDI 2.0 uses with per-note attribute fields and MEI uses with structured extensible metadata, arrived at independently here.

**Microtiming, microtonality, and continuous expression.** Store microtiming as a *deviation* from a notated grid position, never as the position itself: the grid is what phrase and meter analysis need, the deviation is what [expressive-performance](expressive-performance.md)'s rules produce and [groove-and-embodiment](groove-and-embodiment.md)'s feel analysis measures — collapsing them into one raw-seconds number makes both jobs harder. Microtonality has a specific, already-half-solved case in this project's own schema: because `midi` is an integer by convention (not by type), a non-12-TET note needs either a fractional `midi` value or an explicit cents offset carried separately and resolved to `freq` only at the performer; the Performance event's optional `detuneCents?` field is exactly this escape hatch, specified in the schema though not yet exercised by any shipped engine (all five target 12-TET so far), and [tuning-and-scales](tuning-and-scales.md)'s cents-offset tables (slendro, a rast-like approximation, JI ratios) are ready to plug into it. Continuous expression in general (vibrato, swells, filter sweeps) is not representable as a scalar at all; prefer a small parametric shape (rate/depth/delay, as `expr` does) that the performer expands into an automation curve over an explicit breakpoint array — parametric shapes are smaller, and a single "how much" macro (Engine 04's Expression/Bloom/Song sliders) can scale one number instead of rewriting a curve.

**Time bases: ticks, beats, and seconds.** MIDI/SMF ticks, this project's composer-stage float beats, and its performer-stage seconds are three representations of the same underlying quantity chosen for three different jobs: ticks for lossless integer interchange, float beats for a pure, human-legible composer (a phrase transposes and reflows without touching a time value), seconds for the one stage that actually talks to the audio hardware clock. [scheduling-and-timing](scheduling-and-timing.md) covers why the seconds stage specifically needs a lookahead scheduler and a tempo map (the two-clock problem) and is not repeated here. The representational point specific to this page: choose the time base per pipeline stage by what that stage does most — transposition and reflow favor beats, hardware scheduling requires seconds, lossless file interchange favors integer ticks — and convert deliberately at stage boundaries rather than forcing one time base to serve every stage.

## Compact serialization for shareable URL state

*Practical engineering guidance for [comprehensive-site-vision](comprehensive-site-vision.md)'s compact-URL requirement, not a research survey — informed judgment to validate against [control-surfaces-and-user-parameters](control-surfaces-and-user-parameters.md)'s settled parameter list once it exists.*

The current baseline, unchanged across all five shipped engines, is verbose `key=value` via `URLSearchParams` — Engine 01's exact line is `new URLSearchParams({ v: E.VERSION, seed: o.seed, bpm: o.bpm, mode: o.mode, tonic: o.tonic, rev: o.reverb, vol: o.volume })` (`docs/preliminary-tests/engines/01-tonal-classical/index.html`), seven fields already producing a ~60-character tail. Perfectly legible, and fine at this scale, but it grows linearly and verbosely with parameter count — the comprehensive site's ~30-50 Start/Intermediate/Advanced parameters would produce an unwieldy hash under this scheme, and it does not need to be user-decipherable per Tom's stated requirement. Three techniques compose into a better answer:

- **Bit-packing with a version field.** Define, per schema version, an ordered field list with a fixed bit width each — an 8-way genre enum is 3 bits, a slider quantized to 64 steps is 6 bits, a boolean is 1 bit, a seed is 32 bits — concatenate into a bit buffer, and prefix it with a small fixed-width version number that tells the decoder which field list and widths to apply. This is the same idea as SMF's division field or a MIDI 2.0 packet's per-group protocol tag: a short leading tag selects the layout that follows.
- **Varint/delta encoding** for fields whose range is large but typical values are small: a LEB128-style variable-length integer (7 payload bits plus a continuation bit per byte — the same trick SMF delta-times use) avoids reserving worst-case width for the common case, and delta-coding a related sequence before varint-encoding compresses further because deltas cluster near zero (Wikipedia, "LEB128"). Likely more machinery than a single settings payload needs (mostly independent small enums, not a sequence); more relevant if a URL ever carries a short explicit event list rather than just generator settings.
- **A base64url alphabet** (RFC 4648 §5: A–Z, a–z, 0–9, `-`, `_`, no padding) turns the packed byte buffer into URL-safe text at a fixed 4-characters-per-3-bytes ratio with nothing requiring percent-encoding — the right final step regardless of which packing scheme feeds it.

```js
// Illustrative shape only, not shipped code: version (6 bits) + seed (32 bits)
// + N settings at their declared widths, then RFC 4648 base64url, no padding.
function packStateV1(seed, settings) {           // settings: [{ value, width }, …]
  const bits = new BitWriter();
  bits.write(1, 6);                              // schema version selects the field list below
  bits.write(seed >>> 0, 32);
  for (const { value, width } of settings) bits.write(value, width);
  return base64url(bits.toBytes());
}
```

A worked size estimate (arithmetic, not a citation): a 32-bit seed plus roughly 30 mostly-small settings (a mix of enums and quantized sliders averaging perhaps 5 bits each) is about 4 + 30×5/8 ≈ 23 bytes; add a version byte, ≈ 24 bytes; base64url's 4/3 expansion gives ≈ 32 characters — shorter than Engine 01's current 7-field tail, with headroom to roughly double the parameter count before the URL gets unwieldy. Versioning discipline matters more than the encoding: never repurpose a version number or resize/reorder an existing version's field list once any URL carrying it might be live (Tom's shared links are permanent by design), only append new version definitions, and keep every old decoder path reachable indefinitely — the same reason SMF's format byte (0/1/2) and MIDI 2.0's per-group protocol tag let old and new layouts coexist rather than replacing each other.

## Implications for generative engines

1. Update [engine-architecture](engine-architecture.md)'s documented schemas to match shipped practice rather than the reverse: rename `freqHz` to `freq`, and document `beat`/`midi`/`tags`/`weight?`/`expr?` as the Performance event's actual optional fields — all five shipped engines already converged on this shape independently (flagged again in Open questions, since this page cannot edit that file).
2. Default every new per-note capability to a field inside an open, optional, nested object (follow `expr`'s pattern) rather than a new top-level scalar — additive, not breaking, and the same trick MIDI 2.0's per-note attributes and MEI's structured metadata both use.
3. Keep `midi` as an integer-note convention but treat it as 12-TET-*shaped*, not type-enforced: any engine drawing on [tuning-and-scales](tuning-and-scales.md)'s non-12-TET tables should carry an explicit cents offset from the composer stage onward (not only the existing performer-stage `detuneCents?`), so a raga-, maqam-, or gamelan-flavored engine's harmonic material is representable from the start rather than patched on at the last stage.
4. Store microtiming as a deviation from a notated grid position, never as the position itself; losing the grid (raw-seconds-only representations do this) makes both metric analysis and expressive rendering harder.
5. For continuous expression, prefer small parametric shapes over explicit breakpoint curves — smaller, more editable by a style-pack designer, and composable with a single macro control, as Engine 04's sliders already demonstrate.
6. Pick time base by pipeline stage, not globally — beats through the composer, seconds from the performer down, ticks only if a MIDI import/export boundary is ever built.
7. For the comprehensive site's URL state: use a versioned, bit-packed, base64url payload (never `key=value` at 30+ parameters), reserve the version field's width generously up front, and treat [control-surfaces-and-user-parameters](control-surfaces-and-user-parameters.md)'s settled parameter list as the version-1 field-list-and-widths definition once it exists.
8. Keep any future grammar/tree layer (for structure-aware composition, or a [meta-composition-and-style-machines](meta-composition-and-style-machines.md)-style system) as an annotation over the flat event list, not a replacement for it — the flat list is still what the performer, the offline-render harness, and the metrics layer consume.

## Open questions

- Should this project build a Standard MIDI File import/export path (SMF's format is small and well-specified; the Web MIDI API covers live I/O but its cross-browser reach is untested here) for interop with DAWs and notation tools, or is that scope creep for a browser-native, dependency-free project? Not decided; raise in [project-open-questions](project-open-questions.md) if it becomes a real request.
- [engine-architecture](engine-architecture.md)'s documented Note/Performance-event schemas should be reconciled with the shipped-engine convention this page found (`freq` not `freqHz`; `beat`/`midi`/`tags`/`weight?`/`expr?` as real optional fields) — a small, low-risk update for a future session, out of scope for this page's no-other-file-edits constraint.
- The exact bit-width budget for the comprehensive site's v1 URL payload can't be finalized until [control-surfaces-and-user-parameters](control-surfaces-and-user-parameters.md)'s roughly 20 Intermediate-plus-Advanced parameters are settled; the estimate above is a placeholder shape, not a final spec.
- Should a genuinely non-12-TET style pack (raga, maqam, gamelan) get a first-class composer-stage cents field now, speculatively, or only when the first such engine is actually built? [previous-experiments-lessons](previous-experiments-lessons.md)'s caution against speculative abstraction argues for the latter.
