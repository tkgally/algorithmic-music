Site & library architecture Draft for review

The third and final Phase-3b design document: how the comprehensive site is actually organized as files and modules, and — the load-bearing part — the just-in-time composition loop that lets a piece play forever, respond to live control changes without restarting, and still be perfectly reproducible from a seed. It turns the control taxonomy (what the user sets) and the style-vector schema (the data those settings become) into a buildable plan.

Purpose. This is the blueprint a build session works from. It fixes the folder layout, the shared-library module list, the runtime pipeline, the determinism rules, and a walking-skeleton-first build order so that a working site exists early and widens from there. It defers only tuning (the exact preset numbers) and content polish to the build and to later improvement loops.

1 · Scope & constraints

The hard constraints, unchanged from engine-architecture.md and the vision:

  • One self-contained folder, no build step, no server, no network at runtime. The site runs by opening docs/index.html from file:// or GitHub Pages.
  • Vanilla HTML/CSS/JS + Web Audio only. All libraries are original, first-party, and loaded as classic scripts via <script src> into one global namespace — not ES modules, because cross-file import is CORS-blocked under file:// (the settled finding in findings-shared-lib-foundation.md). Instrumental only. Public-domain CC0; the UI is upfront that everything is built by Claude.
  • Determinism: (version, seed, pinned settings) → piece, exactly — the one exception being live tweaks (§6), which is why the shared link is a destination snapshot (schema §8).

2 · Folder layout

The site is docs/index.html plus its own subfolders. The preliminary-test engines and the design docs are separate auxiliary folders that happen to share the docs/ root (they are not part of the site and the site never depends on them).

docs/
  index.html            ← the site: UI shell + <script> includes (the entry point)
  css/
    site.css            ← styling (or inline in index.html)
  lib/                  ← original shared libraries (the reusable core)
    rng.js  transport.js  theory.js  synth.js  fx.js  analysis.js
    generators.js  style.js  compose.js  perform.js  serialize.js
  styles/               ← the eight genre packs (data + any per-genre hooks)
    classical.js  ambient.js  lofi.js  jazz.js  folk.js
    electronic.js  cinematic.js  percussion.js  registry.js
  app.js                ← top-level wiring (URL ↔ state, transport loop, UI events)
  _selftest.html        ← headless offline-render harness (dev-time, like the engines)

  preliminary-tests/    ← the five engines (auxiliary, unchanged)
  design/               ← these design docs (auxiliary)

One global namespace object (say window.AM) carries the modules (AM.rng, AM.style, …), the pattern the engines already use. Everything under docs/lib/ and docs/styles/ is original code; nothing is fetched at runtime.

The composer is the hard, central module. The five engines each proved one composition family. compose unifies them behind one interface (nextUnit returning the next unit's Note events) and picks the family (and its parameters) from the style vector's Material/Evolution fields. It must be incremental — able to produce just the next bar/phrase given the current position and vector — which is what makes §5 and §6 possible. This is the biggest single build task; scope the walking skeleton (§10) around getting one family working end-to-end first.

4 · The runtime pipeline

One directed flow, from a URL to sound:

URL hash ──▶ serialize.decode ──▶ { seed, settings }
                                        │
                              style.sample / style.meld
                                        │
                                 style vector  ◀── (mutable; UI writes here live)
                                        │
   transport.Scheduler ──lookahead──▶ compose.nextUnit(vector, pos) ──▶ Note events
                                        │
                              perform.realize(notes, vector) ──▶ Performance events
                                        │
                                 synth + fx ──▶ Web Audio ──▶ speakers
                                        │
                              (playhead, section) ──▶ UI visualization

The UI reads the seed/settings out and writes them into the URL; the conductor in app.js owns the mutable style vector and the current position, and drives the scheduler.

5 · The just-in-time composition loop

The engines compose the whole piece up front, then play it. The site instead composes a little ahead of the playhead, so the future is always still open. Concretely, the conductor keeps a small horizon (a bar or two) filled ahead of the audio clock:

// conductor tick (called by the scheduler's lookahead)
while (composedThroughBeat < clock.now + HORIZON) {
  const unit = compose.nextUnit(vector, position, streams); // reads the CURRENT vector
  const perf = perform.realize(unit, vector, position);
  synth.schedule(perf, fx.buses);
  position = advance(position, unit);        // bar/phrase/section bookkeeping
  composedThroughBeat += unit.lengthBeats;
}

Because nextUnit reads the current vector each call, anything past the horizon reflects the latest settings — no recomposition of the past, no restart. The horizon is a few hundred ms (bounded by the scheduler lookahead), short enough that changes feel immediate and long enough to stay glitch-free.

6 · Live changes in the loop

A control change mutates the style vector; how fast it is heard is the control's declared response speed (taxonomy §8), which maps onto where in the loop it lands:

SpeedFieldsMechanism
InstantSurface (reverb, brightness, volume, expression amount, width)applied straight to the live fx/synth params (AudioParam ramps); already-scheduled notes ramp in place
Next boundaryMaterial & per-voice (density, mode, palette, swing, meter, lead)the very next compose.nextUnit call reads the new value — lands at the next bar/phrase
Re-planEvolution/form; a genre or blend swaprecompute the remaining trajectory from the current position; a genre/blend swap runs a two-bar crossfade — the outgoing style keeps composing for two bars under a fade-out bus while the incoming style composes on a fade-in bus, then the outgoing bus stops
The two-bar crossfade (Tom's call) is a pair of fx sub-buses with a 2-bar equal-power fade; both styles run their own JIT loop for the overlap, so the seam is a real musical dissolve rather than a cut. Small changes need no crossfade.

Determinism note: with no live changes, the loop is a pure function of (seed, settings) — the RNG streams advance identically every play. A live change perturbs the stream state going forward, so the exact live result depends on when the change happened; that is why the shared link stores the destination snapshot, not the gesture history (schema §8).

7 · Determinism & seeding

  • Named RNG streams (from rng): separate deterministic streams for structure, melody, harmony, rhythm, timbre, and performance, all derived from the one seed — so adding a note to one layer doesn't shift another (the engines already rely on this).
  • The vector gates every draw: compose/perform never call raw random — they sample within the vector's fields and spreads, so the same (seed, vector) yields the same notes.
  • Auto vs pinned: an auto control is filled by the seed within the genre region; a pinned control is fixed and encoded in the URL. Start is almost-all-auto; Advanced pins more.
  • Offline == online: the same pipeline runs under OfflineAudioContext for the render gates, so validation reproduces exactly what a listener hears (the engines' proven approach).

8 · Compact URL state

serialize encodes seed + only the pinned settings (auto fields cost nothing — the seed re-derives them) into a versioned, bit-packed base64url tail: a small version tag, then fixed-width fields (an 8-way genre enum = 3 bits, a 64-step slider = 6 bits, the seed = 32 bits, a second genre + even-balance flag for a meld). The version tag lets old links survive schema growth. The seed is shown, editable, and copy-to-clipboard. The URL always mirrors the current (destination) state (schema §8). Full bit-layout design: music-representation-and-notation.md.

9 · Reusing the engines

The site is not a rewrite — it generalizes proven parts. What lifts directly:

  • Synth voices & fx — the whole accumulated palette (11 percussion voices, 4 expressive voices, drums, Rhodes, bell/pad/drone, keys/strings/bass) and the master chain, merged into one synth/fx.
  • Composition strategies — tonal-classical's phrase/cadence + rounded-ternary form; ambient's incommensurable-loop drift; groove's swing/backbeat/velocity-hierarchy; cantabile's structural-weight → expression; percussion's timeline + interlock + density-arc + the 9 intensity arcs and the meter-parametric grid. Each becomes a compose strategy selected by the style vector.
  • Performer patterns — the phrase-arch tempo/dynamics, structured microtiming, per-section accel/rit, and the intra-note expression layer.
  • UI conventions — the two-level control panel, URL-hash state, piano-roll visualization, and the download-JSON feedback affordance — generalized to the three-mode selector.

The main new work is the style vector + meld + coherence, the incremental compose interface, the three-mode control surface, the compact serialize, and the JIT conductor loop.

10 · Build order & validation

Walking skeleton first: get the whole pipeline working end-to-end with one genre before widening, so a playable site exists early and every later step is additive.

  1. M1 · Skeleton. docs/index.html + app.js + vendored rng/transport/synth/fx; a Play button seeds a trivial one-voice loop from the URL, stops, and round-trips the seed. Proves the loop + audio + URL.
  2. M2 · One genre end-to-end. style with the Classical preset + sample; compose (functional-tonal strategy, incremental) + perform; a full deterministic piece plays from a seed via the JIT loop. Offline-render gate green.
  3. M3 · The three-mode UI + compact URL. Start/Intermediate/Advanced selector; the ~5 Start dials + the ~20 Intermediate controls wired as auto/pin into the vector; serialize compact encoding. Same URL → same piece.
  4. M4 · All eight genres. The remaining seven presets + their compose strategies; the ui-smoke gate plays each. (This is the widest-parallel step — one genre per worker.)
  5. M5 · Two-genre melds. style.meld (per-axis parent selection, even balance, compatibility gates); the multi-select Start buttons.
  6. M6 · Live changes. The three response speeds + the two-bar crossfade for a genre/blend swap; verify no restart and no glitch.
  7. M7 · Invented styles. Novelty-budget sampling + signature rules + the coherence gate, at Intermediate/Advanced.
  8. M8 · Polish. Visualization, feedback JSON, the CC0/authorship footer, and the Advanced invented-parameter controls.

What "the whole build in one session" realistically means: M1–M4 (a working, seeded, shareable three-tier site that composes all eight genres end-to-end, live-playable and offline-validated) is the target floor; M5–M8 land as far as the session reaches, and anything unfinished is a clean follow-up because each milestone is additive. Better a solid M1–M4 than a broken M1–M8.

Validation (reuse the engines' harnesses): an OfflineAudioContext render gate per genre (no clip, no gap, level in band, determinism: same seed → identical PCM) and a headless ui-smoke (the site loads, each genre plays, the URL round-trips, a live change does not restart the transport, 0 console errors). Keep wiki-lint and the node test suite green.

11 · Decisions & open questions

Claude's design calls (Tom can override): the folder layout; the module split (esp. the single incremental compose interface unifying the five composition families); the conductor/JIT-loop shape; and the walking-skeleton build order.

Open, for the build and later loops:

  • Whether compose is one big module with a strategy switch, or one sub-module per family behind a common interface (lean: sub-module per family, thin registry — easier to build in parallel).
  • The exact preset numbers (centers + spreads) for the eight genres — a tuning pass during M2/M4, drawing on the wiki genre pages.
  • How much of the invented-style coherence gate is needed for M-floor vs deferred (presets and melds may not need the full gate; invented styles do).
  • The crossfade's exact equal-power curve and whether its length should scale with tempo (currently a flat two bars).

With this doc, the three Phase-3b design documents are complete (controls → data → architecture). The next step is the build itself (Phase 3c), for which a self-contained build prompt accompanies this doc in the session log.
