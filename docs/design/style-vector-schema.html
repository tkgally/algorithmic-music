Style-vector schema Draft for review

The second design document for the comprehensive site, and the companion to the control taxonomy: the internal style vector — the fielded data structure that every control writes into and the composer reads. It defines the fields and their types, how preset genres, invented styles, and two-genre melds instantiate it, the coherence rules that keep a sampled style listenable, and how it all stays deterministic while responding to live changes. Written by Claude for Tom's asynchronous review.

Decisions folded in (Tom, 2026-07-09). This doc answers the three follow-ups from the taxonomy: two-genre meld balance is even (50/50) for now (§5, flagged provisional — Tom expects to adjust once he hears melds); a live genre/blend swap uses a two-bar crossfade (§7); and the shareable link stores only the destination snapshot (§8). Where a design choice is Claude's own informed judgment rather than Tom's, it says so.

1 · Purpose & scope

The control taxonomy fixed what the user sees and sets. This doc fixes what those settings write into: the style vector the composer consumes.

It decides: the schema's fields (name, type, range, and how each behaves under blending); how preset genres, on-the-fly invented styles, and two-genre melds each produce a concrete vector; the cross-axis coherence constraints; the determinism + live-morph data flow; and a serialization sketch for the shareable link.

It defers: the full shared-library API and function signatures (the site-architecture doc); the exact bit layout of the URL payload (owned by music-representation-and-notation.md); and the build (Phase 3c). It also does not re-list the user controls — see the taxonomy.

Basis: style-invention-and-style-space.md (the axis space, sampling, coherence), meta-composition-and-style-machines.md (the fielded style-machine precedents and the "interpolate quantities, re-choose symbols" rule), hybridization-and-fusion.md (the meld), music-representation-and-notation.md (serialization), and the five preliminary engines' recipe/params as precedent.

2 · The model

Three ideas, each drawn from a documented precedent, define the representation:

  • A style is a fielded vector — a region, not a point. Every style (preset or invented) is the same record of fields; a style stores a center and, per field, a spread (how much the seed may vary it), because a style with no internal spread makes every piece identical (style-invention-and-style-space.md). Learned systems (MusicVAE) confirm style behaves vector-like; consumer systems (Band-in-a-Box .STY, Koan/Wotja) confirm "style = a structured record + a realization engine."
  • A running piece is that vector walking a path. Following Xenakis's "book of screens" and game-audio RTPC state, the piece is a low-dimensional style-state that moves over the form — the natural home for tension shaping and for live control (a knob nudges the current state).
  • Provenance is the only difference between preset, invented, and meld. A preset is a hand-tuned point (the eight Start genres). An invented style is a seeded sample under coherence constraints. A meld is two presets combined per-axis (§5). All three yield the same kind of vector, so one composer core serves them all.

3 · The schema fields

The fields, grouped into four layers (the grouping the style-machine precedents converged on: material · per-voice behavior · evolution · surface). The Type column is load-bearing for both blending and serialization:

categorical — a choice with no perceived midpoint (meter, tuning, mode, form); under a meld it snaps to one parent. ordinal — a small ranked set; blends by weighted choice. continuous — a scalar with a perceptual midpoint (tempo, density, brightness); blends by interpolation. This "interpolate quantities, re-choose symbols" split is the core rule from meta-composition-and-style-machines.md.

4 · Presets, invented styles & the signature bank

  • Preset genre = a hand-tuned center + spreads. Each of the eight Start genres is one filled-in vector: e.g. Classical pins Tuning=12-TET, Harmonic-motion=functional, Meter∈{4/4,3/4}, Form=rounded-ternary/AABA, Tempo≈66–140, and leaves melody/dynamics/expression with wide spreads so seeds differ. These are the reference regions; a future doc/session tunes their exact numbers, drawing on the wiki genre pages the taxonomy cites.
  • Invented style = a seeded sample under constraints. "New style," operationally (EMI-minus-corpus, from meta-composition): draw each field from a genre-plausible prior, spend a novelty budget of 1–3 axes off-convention while holding the rest at perceptual universals, enforce coherence (§6), and attach 2–3 signature rules. Reserved to Intermediate/Advanced per Tom (Start is presets and melds only).
  • Signature rules are what make it read as a style. Cope's EMI showed a small dictionary of recurring fingerprints + a functional grammar is enough to sound like a style. Here the signatures are authored at build time (no corpus to mine): a characteristic interval cell, a rhythmic motto, a cadence/voicing habit — stated in the first phrases and repeated so the listener can latch on within one piece (style-invention-and-style-space.md: teach the style inside the piece).

5 · The two-genre meld Q1

When the user picks two Start genres, the site composes in a blend of the two. The design is grounded in hybridization-and-fusion.md's reading of how real fusions (bossa nova, Cubop, Ethio-jazz, Afrobeat…) are actually built.

5.1 · A meld is per-axis parent selection, not a numeric average

Averaging two whole vectors fails: there is no state "between" 3/4 and 4/4, or between 12-TET and a raga's intonation, and even averaging blendable fields can land between both listenable regions ("fusion or confusion"). Instead, factor the style into layers and source each from a parent:

  • Categorical fields snap to one parent (meter, tuning, pitch alphabet, harmonic-motion type, form, timeline). Pick one parent's value; do not average.
  • Continuous fields blend at the balance weight (tempo, density, harmonic richness, brightness, dynamics, reverb, swing amount).
  • The load-bearing constraint: the rhythmic chassis (meter + groove + tempo-feel) and the harmony/mode come from different parents. This is the one rule every durable historical fusion obeys; which parent supplies the chassis is otherwise a free (seeded) choice — the tempting "dominant parent owns the rhythm" heuristic is not supported by the evidence.
  • One chassis parent, committed for the whole piece (every durable fusion has a single rhythmic frame). Timbre mixes or follows the harmony parent; form comes from one parent or is re-chosen.

5.2 · Balance: even, for now

Tom's call: the default balance is even (50/50). Continuous fields blend at their midpoint; the categorical layers are split fairly between the two parents (each contributes about half the style's layers), subject to the rhythm-≠-harmony rule above.
Provisional. Tom: "I have no idea how this genre meld will actually sound; make a note that adjustments will probably be made in the future." So even-balance is a starting point, not a settled value — expect to revisit it (a seed-varied or user-set balance, per-genre-pair tuning of which parent owns which layer) once there are melds to listen to. The Style blend control (Intermediate/Advanced) already lets the user override the balance.

5.3 · Compatibility gates (reject or repair before committing)

Some parent pairs clash; encode the clashes as gates (hybridization-and-fusion.md):

  • Tuning ↔ harmony: if the mode parent has flexible/non-12-TET intonation, forbid dense tempered harmony — force drone/modal/sparse (as Ethio-jazz does).
  • Meter: if the two cycle lengths disagree, use one parent's meter (snap) or a common cycle; never superimpose two meters.
  • Feel: one swing/microtiming grid per section — a swung and an even parent cannot share a layer.
  • Groove vs rubato: never overlay a free-tempo parent and a metered one; if a pair needs both, sequence them into different sections.

When a gate fails, the incompatible axis yields to the other parent rather than forcing the clash. Melds are capped at two parents (Tom's Start rule; the evidence says three-plus turns to mud). The default operator is a gradient blend (parents dissolved into one texture); a collage operator (parents in alternating sections) is possible later for procedure-clashing pairs.

6 · Coherence constraints

Any vector — preset, invented, or melded — passes the same cross-axis coherence check before it plays (style-invention-and-style-space.md). These are couplings a naive independent sample would violate:

  • Tuning ↔ harmonic motion — drone/modal/static harmony pairs with just intonation; free modulation needs ~12-TET.
  • Tuning ↔ timbre — match the scale to the spectrum (harmonic timbres for small-integer scales), or a chord reads as mistuning.
  • Register ↔ interval spacing — widen harmonic spacing below ~C3 (critical bands are wide low down).
  • Tempo ↔ density — cap events/second to a perceptual ceiling; if tempo is high, thin the texture (keeps perceived complexity in the liking band).
  • Meter ↔ groove — weight syncopation/subdivision to the meter.
  • Form ↔ duration — process/cyclic/drone forms scale long; AABA/verse–chorus want ~1–5 min.

Plus a parseability gate (discrete events, at least one stable reference — tonic/drone/downbeat — and clear grouping cues) and the automated critics the engines can already run (key/hierarchy induction, roughness, information-content band, self-similarity). A vector that fails is repaired (snap the offending axis toward the coupling) or resampled.

7 · Determinism & the live morph Q2

The path from settings to sound: (version, seed, pinned settings) → style vector → per-bar composition → performance → synthesis. The seed seeds named RNG streams; the vector gates and biases every draw, so the whole chain is deterministic (taxonomy principle 2).

Live changes mutate the vector at runtime; the just-in-time composer (taxonomy §8) reads the current vector each bar, so a change lands on not-yet-generated material. The three response speeds map onto the schema layers:

  • Instant — Surface fields (reverb, brightness, dynamics, expression): applied to the next scheduled note and ramped on sustained ones.
  • Next boundary — Material & per-voice fields (density, mode, palette, swing, meter): applied at the next bar/phrase.
  • Re-plan from here — Evolution/form fields, and a genre or blend change: the remaining trajectory is recomputed from the current position.
Tom's call — the live genre/blend swap transition is a two-bar crossfade. When the user changes genre or the meld while playing, the outgoing style and the incoming style overlap for two bars and crossfade, so the seam is musical rather than a hard cut. (Small changes — a slider nudge — need no transition; they are instant or next-boundary.) Grounded in beginnings-endings-and-transitions.md.
Provisional. Tom: "This might be adjusted in the future, too." Two bars is the starting value; the transition length/type may change once live swaps can be auditioned (e.g. tempo-scaled length, or a beat-matched crossfade like a DJ blend for compatible tempos).

8 · Serialization & the shareable link Q3

  • What serializes: the seed plus only the pinned settings (auto fields are re-derived from the seed, so they cost nothing). This becomes a compact, versioned, bit-packed base64url payload — a version tag then fixed-width fields (an 8-way enum = 3 bits, a slider quantized to 64 steps = 6 bits, the seed = 32 bits) — the design owned by music-representation-and-notation.md. The version tag lets old links keep working as the schema grows.
  • Two genres: a meld serializes as the two genre ids + the (currently even) balance + any pinned per-axis parent choices — still a handful of bits.
Tom's call — the shareable link stores only the destination snapshot. The URL always mirrors the current settings + seed, so opening or sharing it reproduces the piece that begins with those settings (played from the top), deterministically. The exact live-morph gesture history is not captured — live tweaking is a real-time performance, and the snapshot reproduces the sound you arrived at. (The "record the automation timeline" option floated earlier is dropped per this decision.)

9 · Three shared generators

The schema's fields feed three small, deterministic generators the shared libraries should provide (from meta-composition-and-style-machines.md) — each seeded, so same-URL-same-piece holds:

  • Interference rhythms (Schillinger) — resultant(a, b) returns a duration-list from two interfering pulse-streams; a cheap cross-rhythm/aksak cell source feeding the Rhythmic-cell supply field.
  • Sieves (Xenakis) — a residue-class builder with union/intersection/complement returning a pitch or onset alphabet; the Pitch-alphabet and Timeline fields' non-diatonic option. Engine 05 already ships a sieve timeline to generalize from.
  • Tendency masks (Koenig) — a [min,max]-over-time sampler that makes any evolving parameter (register, density, dissonance, tempo) move deterministically without repeating; the mechanism behind the Evolution layer and the intensity arc.

The anti-rhetoric lesson carries over (Backus's critique of Schillinger): these generators produce possibility, never value — every field still feeds a constraint/selection layer that carries the musical judgment, or the engine repeats the naive-generativity failure.

10 · Decisions & open questions

Decided (Tom, 2026-07-09):

  1. Meld balance is even (50/50) for now — provisional, adjustments expected once melds can be heard (§5.2).
  2. Live genre/blend swap = a two-bar crossfade — provisional (§7).
  3. The shareable link stores only the destination snapshot (§8).

Claude's design calls in this doc (Tom can override): the four-layer field grouping and each field's type; the per-axis parent-selection meld (rhythm ≠ harmony parent, snap categorical / blend continuous, one committed chassis); the compatibility gates; the coherence gate; and the three shared generators.

Open, for later sessions or listening tests:

  • The exact preset numbers for the eight genres (centers + spreads) — a focused tuning pass.
  • How many signature rules per style are enough to "read as a style," and whether an LLM-authored (no-corpus) signature bank works — needs the evaluation harness.
  • Which fields are truly categorical vs secretly ordinal (does "syncopation amount" or "diatonic↔chromatic" interpolate?) — an A/B once the engine renders midpoints.
  • How many state axes the evolution trajectory needs to carry a satisfying form across genres as different as ambient and groove.
  • The provisional meld balance and swap transition above.

None of these block the build: the schema is concrete enough to start the site's style module and the just-in-time composer. The natural next design doc is the site & library architecture (folder layout, the module list, and the just-in-time composition loop this schema and the live-change model require).
