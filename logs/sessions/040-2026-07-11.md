# Session 040 — 2026-07-11

**Model:** claude-opus-4-8. **Scope:** answer Tom's question about iPhone background audio, and (his choice) document the limitation. No engine or site-behavior change.

## The question

Tom, playing the site from GitHub Pages on his iPhone: *"the music stops when the iPhone screen switches off. Is it possible to have the music continue playing in Safari even when the phone screen is not active?"*

## What I found (research)

**In the codebase:**
- `docs/app.js` `ensureAudio()` creates a bare `AudioContext` (`new Ctx({ latencyHint: 'playback' })`) whose graph runs straight to `ctx.destination`. There is **no `<audio>`/`<video>` element**, no `MediaStreamAudioDestinationNode`, no Media Session — pure Web Audio.
- Playback/composition is driven by a `setTimeout` lookahead scheduler (`docs/lib/transport.js`, the Chris-Wilson two-clocks pattern): the conductor composes only ~0.1 s of events ahead of the audio clock at a time (the just-in-time loop).
- No `visibilitychange`/`resume()` lifecycle handling anywhere in the site.

**Platform behavior (WebKit + MDN, verified 2026-07-11):**
- iOS Safari **suspends/interrupts the AudioContext** when the page is backgrounded or the screen locks — intended behavior (WebKit Bug 237878; desktop Safari does the same on minimize, Bug 231105; standalone PWAs too, Bug 198277). Pure Web Audio therefore stops.
- iOS grants background/lock-screen audio **only to `HTMLMediaElement`** playing media (with Media Session transport controls). A synthesis graph gets no such privilege.

## The answer (given to Tom)

**Not reliably in Safari — and it's an iOS platform restriction, not an app bug.** Crucially, a *generative* engine is blocked **twice**: even if a media-element audio session were obtained, the background JS timer that composes-ahead is frozen by iOS, so the note stream would starve within its lookahead window. Keeping live composition running in the background would need **both** a `MediaStreamAudioDestinationNode`→`<audio>` bridge **and** moving scheduling onto the audio thread (an AudioWorklet) — a real project for uncertain payoff on iOS.

Options laid out, honestly ranked:
1. **MediaStream→`<audio>` bridge** (+ Media Session + resume-on-return): the standard attempt; historically flaky/version-dependent on iOS; still needs worklet scheduling to keep generating. Best-effort, not dependable.
2. **Pre-render a piece to a file, play through `<audio>`**: the only *reliable* background route (with lock-screen controls) — but forfeits live/interactive composition while backgrounded; effectively a different product.
3. **Screen Wake Lock API** (iOS 16.4+): keeps the *screen on* so nothing suspends — battery cost, released on manual lock; partial, not true background.
4. Native wrapper — out of scope (vanilla, `file://`-runnable).

Every engine should at minimum resume on `visibilitychange`/`pageshow` so returning to the tab isn't silent.

## Decision (Tom, via AskUserQuestion)

Offered three next steps — (a) best-effort experiment, (b) reliable pre-render route, (c) just document. **Tom chose (c): document the limitation.** Rationale for not pursuing a workaround now: the reliable path abandons the live composition that is the site's whole point, and the best-effort path has low, on-device-only-verifiable odds — a poor fit for a session that can't test on a real iPhone, and against the project's gate-everything ethos.
