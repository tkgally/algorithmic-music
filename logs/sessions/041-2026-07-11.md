# Session 041 — 2026-07-11

**Scope:** build the background-playback proof-of-concept Tom asked for — a WAV encoder for the rendered buffer, a hidden `<audio>` element, and Media Session wiring, behind a toggle. Additive; the live engine is untouched.

## Context

Sessions 040 established that iOS suspends the live Web Audio engine on screen-lock and grants background audio only to `<audio>`/`<video>` media elements. Tom then asked how a pre-render approach would work; after I described it (render offline → WAV → `<audio>` + Media Session), he asked for a small PoC behind a toggle. This session built it.

## What was built (all additive, behind a **📱 Background** toggle)

**1. A first-party WAV encoder — `docs/lib/wav.js`** (new, UMD-lite, CC0, node-testable like `rng.js`/`transport.js`). `encodeWav(audioBufferLike)` writes a complete 16-bit PCM RIFF/WAVE file (`Uint8Array`) from anything shaped like an `AudioBuffer` (`numberOfChannels`/`sampleRate`/`length`/`getChannelData`). The site already renders a whole piece deterministically (`app.js` `renderOffline` → stereo `AudioBuffer` at 44.1 kHz, the same path the self-test/render gates use), so this is the only missing piece to get a file.
   - **Why WAV:** it's the only container the browser can produce in **pure vanilla JS with no dependency and no build step**, and it plays on every iOS version. MP3/AAC would need a vendored WASM encoder (against the project's dependency-free rule); Ogg/Opus aren't reliable on iOS. Cost is size (~172 KB/s at 16-bit stereo 44.1 kHz → ~11 MB for a 67 s piece), which is fine for transient, in-memory, one-piece-at-a-time use.
   - Node tests (`experiments/tests/wav.test.js`, +5): header fields, frame-major interleave with exact values, clamp+round quantization (incl. the `Math.round` tie-toward-+∞ asymmetry: `-0.5 → -16383`), mono, all-zero silence.

**2. The playback flow — `app.js` `playRendered()`** (new background-mode section, ~120 lines). Render the current state → `encodeWav` → `Blob` → `URL.createObjectURL` → an off-screen `<audio id="bgAudio">`.
   - **The iOS gesture trick:** media must start inside a user tap, but rendering is async and the tap "expires." So on Play we immediately play a **looping-silence WAV** in-gesture to claim the media session, then swap the real file's object URL in once the render resolves (allowed because the element already played from a gesture). A `seq` counter drops superseded/stopped renders.
   - `bgStop()` pauses, clears the src, revokes the object URL, and clears the media session; it's folded into `stopAll` so switching between live and background modes tears down cleanly. The Play button, dice, seed-change, and style-swap handlers are background-aware (a style/seed change re-renders).

**3. Media Session wiring** (`setupMediaSession`): metadata (title = piece name, artist = seed, a generated canvas artwork so the lock-screen card isn't blank) + `play`/`pause`/`nexttrack`/`previoustrack` handlers. `nexttrack` reseeds and re-renders; continuous play re-renders on the `<audio>` `ended` event. All guarded (`'mediaSession' in navigator`, `typeof MediaMetadata`).

**4. UI:** a `📱 Background` toggle in the transport bar (reuses the `.loopBtn` toggle style), the off-screen `<audio>` element (kept in the DOM, positioned off-screen rather than `display:none` so it stays "rendered"), and `lib/wav.js` added to the script tags. Dev/smoke hooks on `AMApp`: `playRendered`, a bounded `renderWavURL({capUnits})`, and `bgMode`/`bgActive`.

## What it trades (documented for Tom)

A rendered piece is a **fixed file**, so the live no-restart setting changes don't apply while it plays — changing style/seed **re-renders** (a brief "composing…"). It's a *mode*, not a replacement for the live engine. Nothing is serialized into the URL (`bgMode` is a session toggle like `continuous`), so shared links and the URL layout are unchanged, and the live conductor and determinism are untouched.
