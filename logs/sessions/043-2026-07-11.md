# Session 043 — 2026-07-11

**Scope:** eliminate the gap between tracks in continuous background play by pre-rendering the next track while the current one plays. Additive, background-only.

## Tom's report

> "Let's go ahead and do the rendering of the next track while the current one plays. I listened to some tracks on Continuous playback, and there was quite a long pause when the song length was set to medium. It's probably even longer with long. Rendering in advance would be a big help."

The pause was the next track's render happening *at* the boundary (in the old `advanceRendered` → `playRendered` path): each track ended, then the next rendered from scratch (seconds, scaling with piece length) before playing.

## The change — double-buffering (`docs/app.js`, background section only)

While the current track plays, render the next one ahead of time and hold it ready; the boundary hand-off then just swaps the ready blob in.

- **`prepareNext()`** — renders the next track (a fresh seed, current `sel`/`controls`) to a WAV blob and stores it in `prepared = { seed, url, info }`. Called at the end of `playRendered` (once the current track is playing) and at the end of each `advanceRendered` hand-off, and when Continuous is toggled on during background playback. The render runs off the main thread, so the playing `<audio>` is unaffected.
- **`advanceRendered()`** — if `prepared` is ready, swap it into the element and play **instantly** (via a shared `startTrack` helper), then `prepareNext()` for the track after. If it isn't ready (the first transition, or a render slower than the track), fall back to the old render-on-demand path — a brief gap, then continuous is caught up.
- **Lifecycle / correctness.** A `prepareToken` (bumped on each new prepare and on stop/discard) plus the session counter `bg.seq` (bumped by `playRendered`/`bgStop`) make any **superseded** prepare discard itself and revoke its blob URL: a stop, or a live style/seed change (which routes through `playRendered` → `stopAll` → `bgStop`), invalidates an in-flight prepare so a stale next-track is never played. `bgStop` also drops `prepared`. Toggling Continuous off during background playback calls `discardPrepared()` (frees the blob); toggling it on calls `prepareNext()`.
- **Why one `<audio>` element, not two.** A second element preloaded for true zero-latency was considered and rejected: on iOS a fresh element lacks the user-gesture activation that the single reused element already has, so playing it at the boundary could be blocked. Reusing the one unlocked element (swap `src`, `play()`) is the safe path; the residual cost is at most a sub-second WAV decode, versus the multi-second render it replaces.
