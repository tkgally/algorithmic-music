# Session 046 — 2026-07-12

**Model:** Claude Opus 4.8 · **Branch:** `claude/slider-auto-reset-19nb3l` (from `main` at the merged session-045 state)

## What Tom asked

A single focused UI tweak on the live v1.0.0 player, described with three screenshots:

- **State 1** — app just opened: every control on *auto*, the sliders inactive with the white thumb on the far left.
- **State 2** — a slider dragged: its AUTO chip goes inactive and the track fills (amber) up to the thumb. *(Correct — the intended pinned look.)*
- **State 3** — the slider's AUTO chip then pressed: the slider **still showed the dragged position** (amber fill up to the thumb).

Tom's read of the intended operation: after AUTO, the slider should return to **State 1** — the rest position, showing no manual setting. Fix it.

## Diagnosis

He's right, and the fix is one line. In `docs/app.js`, `syncCtl(id)` is the single function that reconciles a control's DOM with the pin state (`state.controls[id]`). Its unpinned (auto) branch handled dropdowns (set to the italic "choose…" placeholder) but for sliders only cleared the readout — it never reset the `<input type=range>` value:

```js
} else {
  // Auto: a dropdown shows the italic "choose…" placeholder (value "");
  // a slider keeps its thumb position but shows no value in the readout.
  if (isSelect) { input.value = ''; input.classList.add('placeheld'); }
  val.textContent = '';
}
```

The range inputs are painted with `accent-color: var(--accent)` (`css/site.css` line 145), so the browser fills the track from the minimum up to the thumb. A left-behind non-zero value therefore keeps the amber "filled" look even though the label color, readout, and AUTO chip had all correctly reverted (those are driven by the `.pinned` class toggle a few lines up). So the control was *logically* on auto but *visually* still read as manually set. This was a deliberate-but-wrong choice from session 045 (its findings note even recorded "sliders were left as-is").

## The change

One added line in the unpinned branch of `syncCtl` — reset a range input to its rest value:

```js
} else {
  // Auto: a dropdown shows the italic "choose…" placeholder (value "");
  // a slider returns to its rest position (thumb far left, no accent fill),
  // matching the initial open state so nothing reads as manually set.
  if (isSelect) { input.value = ''; input.classList.add('placeheld'); }
  else input.value = '0';
  val.textContent = '';
}
```

`'0'` is exactly the value `controlRow` builds every range with (`min: '0'`, `value: '0'`), so State 3 becomes byte-identical to State 1. Because `syncCtl` is the shared reconciler, this also fixes the **Reset** button (`resetControls` → `syncAllControls` → `syncCtl`): Reset now returns all sliders to rest, not just clears their chrome.

`SITE_VERSION` bumped **1.0.0 → 1.0.1** (patch — a small user-visible fix, per the `CLAUDE.md` convention).

## Notes / decisions

- **Presentation-only.** The edit is the `else` branch of one UI-controller function plus the version constant. The audio engine (`docs/lib/**`, `docs/styles/**`), determinism, and the compact-URL layout are untouched — `state.controls` already deleted the pin on un-pin; only the slider's *visual* was lagging.
- **Affordance symmetry is the principle:** a control on auto must look the same whether it was never touched or pinned-then-released. Session 045 established this for dropdowns; this session extends it to sliders. The reusable lesson (filed in the wiki): **`accent-color` sliders carry visible state in their value, so "return to auto" means resetting the value, not just the surrounding chrome.**
