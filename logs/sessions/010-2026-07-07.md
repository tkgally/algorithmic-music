### Shared-library foundation prototype (`experiments/lib/`)

Prototyped the two cheapest foundation modules from `wiki/shared-libraries.md`, as original dependency-free code reimplementing published algorithms (mulberry32, Bjorklund/Toussaint Euclidean rhythm, Voss–McCartney pink noise, Wilson's two-clock scheduling) — no source copied.

- **`rng.js`** — `Rng` (seeded mulberry32) with float/int/bool/pick/weighted/shuffle/gaussian, named independent sub-streams (`rng.stream("bass")`), a `PinkNoise` (1/f) generator, and standalone `euclid(pulses, steps, rotation)` + `onsets()`.
- **`transport.js`** — `MusicClock` (pure beats↔seconds/bars) and `Scheduler` (lookahead "two clocks" with injected `now`/`setTimer`/`clearTimer`).
- **`tests/`** — a tiny dependency-free runner + suites; `node experiments/tests/run.js` → **25 passing assertions**, including an exhaustive maximal-evenness proof of the Euclidean output for all E(k,n), 1 ≤ k ≤ n ≤ 32 (also spot-checked to 48), deterministic named streams, and correct lookahead-window scheduling driven by a fake clock.
- **`demos/euclid-transport.html`** — a `file://`-runnable demo wiring both modules to real Web Audio; **`README.md`** documents layout, how to run, and provenance.

Two decisions were settled by the prototype and filed in the new **`wiki/findings-shared-lib-foundation.md`**:

- **Format = dual-format (UMD-lite) classic scripts, not ES modules.** A vendored library must load from `file://`; a classic `<script src>` does, but a cross-file ES-module `import` is CORS-blocked under `file://`. So each module is one file that works as both a Node `require` and a browser global (`window.AM.*`) — validated both ways (Node tests + a VM-sandbox check of the browser-global branch). This resolves the plan's granularity/format open question.
- **Injected dependencies** (pass the timing source / `AudioContext` in; no globals) is the standing pattern for the Web-Audio libraries and is what makes them headless-testable.

Threaded the results back into `shared-libraries.md` (a prototype-status section + resolved open question), `rhythm-and-meter.md` (the validated Bjorklund implementation, confirming its own E(5,16) = 3-3-3-3-4 example), and `scheduling-and-timing.md` (Related link). ## Decisions / assumptions

- **Ran the two-part scope in parallel** rather than sequentially, since the branch name encoded both and they don't compete for the same resources. Kept the evaluation edits batched until all four subagent reports were in, to apply them coherently.
- **Switched the Euclidean implementation to canonical remainder-recursion Bjorklund** (from a first-draft interleave method) for correctness confidence; validated with an exhaustive rotation-invariant maximal-evenness property test. The initial hardcoded test expectations for E(3,4)/E(2,3) encoded Toussaint's *traditional* rotation, not Bjorklund's normalized output — corrected the tests, not the code, and exposed rotation as a parameter (documented that traditional presentations are specific rotations).
- **Marked the findings page `reviewed`** (not `draft`) because its claims are backed by a reproducible in-repo test suite (`node experiments/tests/run.js`) — a stronger verification than source-fetching.
- **Did not build the OfflineAudioContext audio-timing harness this session.** Playwright is not installed as a node module (the project is dependency-free) and a real-audio harness is more than "the two cheapest modules." The Node suite proves the library *logic*; real audio timing is explicitly deferred to queue item 7. The demo is the first target for it.
- Trusted the verification subagents' sourced corrections (they fetched the cited sources and, for corpus-analysis, read the actual PDF rather than the summarizer — which is how the Essen hallucination was caught).
