## The fix â€” graceful track endings

**Root cause.** A rendered file's length is `musicSec + tail` (`renderOffline`). For `fade` endings a fade is scheduled, but for **non-fade endings (cadence/stop/ringout) nothing shaped the file boundary** â€” so the file could end while the reverb tail was still ringing, i.e. a hard digital cut. The *live* engine hides this in continuous play because `advancePiece` starts the next piece so its attack overlaps the outgoing tail; a **standalone rendered file has nothing after it**, so the cut is fully exposed against the following silence â€” exactly what Tom heard.

**Fix** (`docs/app.js` `renderOffline`, background-only â€” the live conductor doesn't use this function, and the render gate uses `_selftest.html`'s separate `renderOnce`, so nothing else is affected):
- Fuller reverb tail so the file rings out to near-silence before it ends: `tail = 0.6 + reverbSecondsÂ·1.1` (was `0.4 + reverbSecondsÂ·0.8`).
- A short **safety fade to true silence over the final ~0.8 s** for non-fade endings (an exponential ramp to ~0.0008 ending 20 ms before the file end). It only shapes the very end once the music has essentially finished â€” it does not soften a composed cadence, it just guarantees a clean boundary.

**Verified** in a real browser (`scratchpad/tail-verify.mjs`, not committed): three classical seeds (all `cadence` endings) now have **peak 0 in their last 50 ms and 200 ms**, while whole-file peak stays ~0.5â€“0.7 and the level at the music-end is already low (~0.02â€“0.05) thanks to the fuller tail. So each track rings out to silence instead of cutting.

## Answers to Tom's questions (given in chat; captured in the wiki)

**Android:** the *live*-engine background failure is not iOS-only â€” Android Chrome also throttles/freezes background JS timers, which would starve the compose-ahead scheduler, so the live engine likely stalls there too (iOS is just the strictest, hard-suspending the `AudioContext`). But the **pre-render Background mode works on both iOS and Android** â€” Android Chrome is more permissive with media-element background playback, and the Media Session wiring already gives Android a media notification / lock-screen card. Caveat: Android battery-optimization settings can still interfere on some devices. Sources: [WebKit Bug 237878], [spf.io on Android background audio], [Chromium issue 41452188] (device-specific WebRTC background drops).

**Exposure / device detection:** yes, the site can offer Background conditionally. Because it helps on **both** iOS and Android (not iOS-only), the natural scope is **mobile/touch devices**, best detected by capability â€” `matchMedia('(pointer: coarse)').matches` + `navigator.maxTouchPoints > 0` â€” rather than user-agent sniffing (iPadOS reports as desktop Safari and defeats UA checks). On desktop it's rarely needed. **I did not implement any UI gating** â€” Tom explicitly wants to do the interface design with me in a day or two, so this is his call; the recommendation is recorded to inform it.
